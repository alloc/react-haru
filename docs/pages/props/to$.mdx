The `to` prop decides the final value of an animation.

It's **not** "set in stone". For example, as your component re-renders, it can update the `to` prop, and the animation will seamlessly adjust itself (assuming you used spring physics).

### Acceptable values

When `to` is defined in a `Controller` update, it **must** be an object, array, or function. This is how `useSpring` and other hooks expect their `to` prop to be.

```ts
useSpring({
  to: { x: 0, y: 0 },
})
```

See the [Async animations](#async-animations) section if you're curious how `to` as an array or function works. It's further down this page, so you can just keep reading, too!

If you're using the `.start` method of `SpringValue`, the `to` prop cannot be an object, unless the `SpringValue` has a defined `key` property. Otherwise, you should define `to` as the goal value.

```ts
spring.start({ to: 1 })
```

### Inferred values

The `to` object can be inferred from the unreserved props (without sacrificing type safety). In other words, you can skip nesting your `to` values in an object entirely!

```ts
const { x, y } = useSpring({ x: 0, y: 0 })
```

### Using fluid values

With the power of [fluid values](/fluids), your animation can react to dynamic values, like other [`SpringValue`](/objects/SpringValue) objects or even [interpolated values](/interpolation).

When `to` is a fluid value, the animation's final value depends on the fluid value. This can be extremely useful for advanced use cases. If the fluid value changes after the animation is finished, a new animation will start automatically.

## Override delayed updates

If you update the `to` prop before a delayed update is applied, the delayed update will be cancelled automatically. For example, this is extremely useful when animating in response to `mouseenter` and `mouseleave` DOM events, and you're using a `delay` for `mouseleave` animation but none for `mouseenter` animation. With zero effort on your part, the `mouseleave` animation will never accidentally override the `mouseenter` animation.

# Async animations

An **async animation** is a group of spring updates applied by the same `to` prop. These updates can be grouped in an array or generated by a function on-demand.

Async animations continue until finished, but they can be interrupted with the `.stop` method, the [`cancel`](/props/cancel) prop, or another async animation.

You **must** provide a `from` prop when using async animations.

## Sequences

When `to` is an array of updates, it will apply those updates in order as a "sequence". The first update is passed to the [`.start`](/objects/Controller#start) method, and the next update won't be applied until the first promise resolves. If any promise in the sequence resolves with `finished: false`, the sequence is interrupted.

```ts
useSpring({
  from: { color: 'white' },
  to: [{ color: 'red' }, { color: 'blue' }, { color: 'green' }],
})
```

## Scripts

When `to` is a function, it receives an `animate` function. The sky is the limit with this approach. Your `to` function can use async-await with any promise, including the promises returned by `animate` calls.

```ts
useSpring({
  from: { color: 'white' },
  to: async animate => {
    // TODO
  },
})
```

Any [default props](/props#default-props) defined with the `to` function are inherited by your `animate` calls,
but outside updates will **not** inherit those default props.

### Nesting

You can even start an async animation from inside a `to` function.

```ts
useSpring({
  to: async animate => {
    await animate({
      to: async animate => {},
    })
  },
})
```
